# SYNAR — Engineering Challenges and Solutions

## 1. Streaming delivered full responses instead of real chunks

**Symptoms**
- responses appeared after long delay as one full block
- chunked UX was broken
- stream behavior was inconsistent through Ollama

**Root causes**
- weak stream normalization
- missing buffer layer
- incorrect chunk-event handling

**Architectural response**
- introduced stream adapter
- added buffer layer
- normalized chunk structure
- treated streaming as a dedicated subsystem instead of a UI side effect

---

## 2. Duplicate output after streaming

**Symptoms**
- text repeated
- final reply appeared again after stream completed

**Root causes**
- secondary final reply push from orchestration layer
- unsafe tail reconstruction logic
- overlapping output authority between stream and final reply

**Architectural response**
- removed secondary final response push
- removed tail reconstruction behavior
- enforced stream as the single source of truth for live output
- kept final state authoritative for storage / reload only

---

## 3. Long answers were cut off

**Symptoms**
- incomplete long replies
- answers ended early
- continuation metadata did not match actual truncation state

**Root causes**
- low generation budget in FAST paths
- long requests routed into lightweight mode
- weak cutoff detection

**Architectural response**
- improved continuation conditions
- separated long-output handling from lightweight budget logic
- strengthened route selection for long and analytical requests

---

## 4. Routing logic sent the wrong requests to FAST mode

**Symptoms**
- deep or long prompts were answered by the wrong path
- route decisions were inconsistent
- multilingual detection was fragile

**Root causes**
- brittle regex logic
- word-boundary behavior breaking on Cyrillic and multilingual inputs
- weak distinction between “long answer” and “deep reasoning”

**Architectural response**
- split `long_output_intent` and `deep_analysis_intent`
- shifted toward `intent → policy → route`
- moved away from fragile regex-first routing
- improved multilingual direction

---

## 5. Infrastructure and AI logic were drifting into one file

**Symptoms**
- growing server entrypoint
- harder debugging and riskier patching
- architecture becoming fragile under quick fixes

**Root causes**
- monolithic pressure in `server.js`
- too much runtime logic sitting in infrastructure layer

**Architectural response**
- moved core behavior toward `core/synar-engine.js`
- preserved the rule that infrastructure stays separate from AI logic
- treated modularization as an architectural requirement, not cleanup

---

## 6. Model behavior leaked into system identity and control

**Symptoms**
- model drift such as “I am SYNAR” behavior
- internal role/debug answers generated by the model instead of system layer
- blurred boundary between system state and model output

**Root causes**
- missing server-side authority for control/debug answers
- weak separation between policy/control and generation

**Architectural response**
- moved toward creator/admin-aware control handling
- defined that system-level answers should come from SYNAR control logic, not model improvisation
- reinforced principle: the model generates, SYNAR decides

---

## 7. Memory needed boundaries, not just more context

**Symptoms**
- memory bleed into unrelated answers
- context contamination risk

**Root causes**
- insufficient separation between system prompt, memory, and user content
- lack of memory isolation rules

**Architectural response**
- planned explicit Memory Layer
- defined prompt separation requirements
- treated memory as a controlled subsystem, not a loose prompt append

---

## Why these cases matter

These cases show that SYNAR was developed as a real AI systems project shaped by failure analysis, system control, and architecture correction — not as a static chatbot demo.
